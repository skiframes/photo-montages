What is Skiframes?

Skiframes is an open-source video analytics platform for alpine ski racing. It automatically generates stop-motion photo montages from race video, showing an athlete's full run through a course section in a single image.

The technique is inspired by Ron LeMaster's ski photography from Ultimate Skiing, where multiple exposures of a skier are composited onto a single background to reveal body position, line, and timing through a gate sequence.

How It Works

Cameras positioned along the race course record video at 30 frames per second. Software detects each skier entering and leaving the camera's field of view, extracts the relevant frames, and composites them into a single montage image. The result shows the skier's progression through 4-8 gates in one picture.

Camera on course          Software detects run          Photo montage generated
       |                         |                              |
  [30fps video] ---> [Frame differencing] ---> [Stop-motion composite]
       |                         |                              |
  Records all day     Triggers on motion         One image per athlete
                      in trigger zones           per camera section

Understanding Results

Official vs. Unofficial Times

The Rank and Time columns show official results from the Vola electronic timing system (wired timing wands at start and finish). These are the authoritative race results.

The Section times (under each camera section) are unofficial estimates derived from video analysis. They show how long the athlete was visible in each camera's field of view. Accuracy is approximately ±0.08 seconds.

Section times are useful as a coaching tool to compare athletes' speed through different parts of the course, but they should not be used for any official purpose.

FPS Variants Explained

Each photo montage is generated at multiple frame rates:

RateFrames UsedBest For
5.0 img/sFewerCleaner individual body positions, less overlap
6.0 img/sMedium (default)Good balance of detail and clarity
7.0 img/sMoreSmoother motion trail, more continuous flow

The default is 6.0 img/s. Use the FPS buttons in the lightbox to experiment and find the best visualization for each athlete.

Camera Setup

Hardware

The system uses three Reolink RLC-823S2 cameras (2K resolution, 30fps, 5Mbps RTSP streams). Each camera covers a different section of the race course.

Cameras connect to the edge compute device via ethernet or WiFi. The edge device connects to the internet via a 5G uplink (Teltonika RUTM50 router) for remote access and S3 uploads.

Camera Placement

Position cameras to capture 2-8 gates each. Key considerations:

  • Field of view — ~55° horizontal. Position far enough to capture the full gate sequence.
  • Angle — Looking uphill or across the slope. Avoid looking directly downhill (skiers approach head-on with little visible body movement).
  • Stability — Use heavy-duty tripods. Wind and vibration cause misaligned composites.
  • Overlap — Slight gate overlap between cameras is fine; gaps mean missed coverage.

Calibration UI

The calibration interface is a mobile-first web app served by the Flask server on the edge device. Coaches access it from their phone while on the mountain.

Workflow

  1. Select camera — Choose from R1, R2, R3, or Axis in the dropdown
  2. Select input — Live RTSP stream or recorded video file
  3. Grab a frame — Capture a representative frame from the video
  4. Draw START zone — Rectangle near the top of frame where skiers enter
  5. Draw END zone — Rectangle near the bottom where skiers exit
  6. Configure session — Set type (race/training/test), group (U10-Masters), end time
  7. Save & start — Saves config JSON and begins detection

Trigger Zone Concept

The gates being filmed are between the START and END zones. When a skier enters the START zone, the system begins tracking. When they exit the END zone, the run is complete and a montage is generated.

Tuning Detection

Detection sensitivity can be adjusted in the calibration UI. Changes take effect immediately (hot-reload every 2 seconds) without restarting detection.

Parameters

ParameterDefaultRangeEffect
detection_threshold 25 0-255 Minimum pixel difference to count as changed. Higher = less sensitive. Increase if shadows or snow spray cause false triggers.
min_pixel_change_pct 5.0% 0-100% Minimum percentage of zone pixels that must change. Higher = requires more movement. Increase for noisy/windy conditions.
min_brightness 100 0-255 Changed pixels must be at least this bright. Filters shadows (dark) while keeping skiers (bright against snow). Key for sunny days.
cooldown_seconds 3 0-30 Minimum time between consecutive runs. Prevents double-triggers from same skier.

Troubleshooting

  • Too many false positives — Increase detection_threshold to 35-45, increase min_brightness to 120-150
  • Missing runs — Decrease detection_threshold to 15-20, decrease min_pixel_change_pct to 3%
  • Shadow triggers — Increase min_brightness to 130+ (shadows are darker than skiers)
  • Wind/snow triggers — Increase min_pixel_change_pct to 8-10%
  • Double-counting same skier — Increase cooldown_seconds to 5-8
How detection parameters interact

For each video frame, the detection engine:

  1. Converts frame to grayscale
  2. Computes absolute difference with previous frame: diff = |frame_n - frame_n-1|
  3. Crops to the trigger zone region
  4. Thresholds: pixels where diff > detection_threshold are "changed"
  5. Brightness filter: only keeps changed pixels where brightness > min_brightness
  6. Counts remaining changed pixels as percentage of zone area
  7. If percentage > min_pixel_change_pct, the zone is triggered

The brightness filter is the key innovation for ski racing: snow is bright (200+), skiers are moderately bright (120-200), but shadows are dark (30-80). By requiring changed pixels to be bright, shadows are filtered out while skiers pass through.

Detection Engine

The detection engine (detection.py) monitors video frames for skier motion using frame differencing in configurable trigger zones.

Core Algorithm

  1. Convert each frame to grayscale
  2. Compute absolute pixel difference with previous frame
  3. Crop difference image to the trigger zone rectangle
  4. Apply threshold: pixels with diff > detection_threshold are "changed"
  5. Apply brightness filter: keep only changed pixels brighter than min_brightness
  6. Count changed pixels as percentage of zone area
  7. If percentage exceeds min_pixel_change_pct, zone is triggered

State Machine

  IDLE  ---(START zone triggered)--->  TRACKING
    ^                                      |
    |                                      |
    +----(cooldown expires)----  COMPLETE <-+--- (END zone triggered)
                                    |           (or timeout)
                                    v
                              Generate montage

Key Classes

  • Zone(x, y, w, h) — Rectangle defining a trigger area on the frame
  • FrameBuffer — Ring buffer (deque) holding the last N seconds of frames for pre-trigger capture
  • DetectionConfig — All parameters loaded from JSON, supports hot-reload (polled every 2 seconds)
  • DetectionEngine — Main class with process_frame(), run_on_video(), run_on_rtsp()
  • Run — Data class holding run_number, timestamps, and captured frames
FrameBuffer and pre-trigger capture

The FrameBuffer continuously stores the most recent frames (default: 2 seconds worth). When the START zone triggers, the buffered frames are included in the run, capturing the skier's approach before the trigger fired. This ensures no part of the run is missed due to detection latency.

# Simplified frame buffer logic
buffer = deque(maxlen=fps * pre_buffer_seconds)

def process_frame(frame):
    buffer.append(frame)
    if detect_motion_in_zone(frame, start_zone):
        # Include buffered frames from BEFORE the trigger
        run_frames = list(buffer) + [frame]
        start_tracking(run_frames)
Stream Manager for shared connections

When multiple detection sessions run on the same camera (e.g., different coaches monitoring different zones), the Stream Manager (stream_manager.py) maintains a single RTSP connection and distributes frames to all registered sessions. This avoids duplicate decode overhead.

Each session gets its own copy of each frame to prevent race conditions, and maintains independent detection state (zones, thresholds, run tracking).

Montage Generation

The montage generator (montage.py) creates stop-motion photo composites from the frames captured during a run.

Pixel-Wise Minimum Algorithm

The core technique is remarkably simple: for each pixel position, take the darkest value across all frames. Since the snow background is bright (white) and the skier is darker, the skier's silhouette from every frame is preserved while the snow background remains clean.

  Frame 1        Frame 2        Frame 3        Composite
  ┌────────┐    ┌────────┐    ┌────────┐    ┌────────────┐
  │  ○     │    │    ○   │    │      ○ │    │  ○  ○   ○  │
  │  |     │    │    |   │    │      | │    │  |  |   |  │
  │ / \    │    │   / \  │    │    / \ │ => │ / \/ \ / \ │
  │ snow   │    │ snow   │    │ snow   │    │ snow        │
  └────────┘    └────────┘    └────────┘    └────────────┘

  np.minimum(frame1, frame2, frame3, ...)
  Keeps darkest pixel at each position

Processing Pipeline

  1. Frame selection — Subsample captured frames to target FPS (e.g., 30fps capture → 6fps montage = every 5th frame)
  2. Crop — Compute bounding box from START/END zones with asymmetric padding (30% left, 50% right, 10% vertical) to capture the full ski path
  3. Bilateral filter — Edge-preserving noise reduction (cv2.bilateralFilter(f, 5, 50, 50)) applied to each frame before compositing. Reduces noise by ~50% while preserving edge sharpness.
  4. Composite — Iterative pixel-wise minimum: result = np.minimum(result, next_frame)
  5. Output — Generate thumbnail (~800px, JPEG 85%) and full-resolution (native crop, JPEG 95%)
Why bilateral filter instead of Gaussian blur?

Gaussian blur smooths everything equally, including edges. Bilateral filtering smooths flat areas (reducing sensor noise in the snow) while preserving sharp edges (keeping the skier's outline crisp). Quality comparison:

MetricRaw MinimumWith Bilateral Filter
Noise level2213 (41% reduction)
Edge sharpness30172873 (5% reduction)

The trade-off is worthwhile: nearly half the noise removed with minimal sharpness loss.

FPS variant generation

Each run generates montages at 5 different frame rates (5.0, 5.5, 6.0, 6.5, 7.0 images per second). This is done by subsampling the captured frames differently:

# 30fps video, 2.5-second run = 75 frames captured
# At 5.0 img/s: use every 6th frame = 12 frames in composite
# At 7.0 img/s: use every ~4th frame = 17 frames in composite

def select_frames_by_fps(frames, source_fps, target_fps):
    interval = source_fps / target_fps
    indices = [int(i * interval) for i in range(int(len(frames) / interval))]
    return [frames[i] for i in indices if i < len(frames)]

More frames = smoother motion trail but more overlap between positions. Fewer frames = cleaner individual positions but less continuous flow. The default (6.0 img/s) is a good middle ground.

3D Terrain Pipeline

The 3D course visualization renders the actual race terrain using a digital elevation model and satellite imagery in Three.js.

Data Sources

DataSourceResolutionFile
ElevationUSGS 1m DEM (LiDAR)1 meter/pixelterrain_heightmap.bin
Satellite imageryEsri World Imagery~0.3m/pixel (zoom 19)terrain_texture.png
Terrain metadataExtracted from DEM—terrain_meta.json

Rendering Pipeline

  1. Load heightmap — Binary file of 16-bit elevation values, 800×800 pixels
  2. Create geometry — Three.js PlaneGeometry with vertices displaced by elevation values
  3. Apply texture — Satellite or hillshade image mapped onto the terrain surface
  4. Add course overlay — Gate spheres at GPS positions, projected onto terrain surface
  5. Add section shading — Colored polygon meshes for each camera's coverage area
  6. Lighting — Ambient (0.55) + directional sun (0.85) + fill light (0.3)
GPS to 3D coordinate mapping

GPS coordinates (latitude, longitude, elevation) are converted to Three.js world coordinates using the terrain metadata:

function gpsTo3D(lat, lon, elev) {
  // Normalize to terrain grid position (0..width, 0..height)
  const x = (lon - bounds.west) / (bounds.east - bounds.west) * terrainWidth;
  const z = (lat - bounds.north) / (bounds.south - bounds.north) * terrainHeight;

  // Elevation scaled by terrain metadata
  const y = (elev - elevMin) * elevScale;

  return { x, y, z };
}

The terrain covers approximately 800×800 meters centered on the race course, with elevation ranging from ~184m to ~330m above sea level.

Terrain extraction process

The extract_terrain.py script processes a USGS GeoTIFF into web-ready formats:

  1. Download 1m DEM tile from USGS National Map
  2. Crop to area of interest (race course + buffer)
  3. Sample elevation at 2m intervals (reducing 800m extent to 400 samples)
  4. Quantize to 16-bit unsigned integers
  5. Write binary heightmap + metadata JSON
  6. Generate hillshade texture using sun angle computation
  7. Fetch satellite tiles from Esri and composite into a single texture
